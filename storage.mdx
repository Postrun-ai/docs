---
title: "Storage"
description: "Store your social and ads data on Postrun and query it back — no database of your own."
---

Postrun is the **system of record** for your social and ads domain. Your
profiles, posts, schedules, and media live with us, and you read them back
through one API — so you don't run a parallel database, write sync jobs, or stand
up backend routes that exist only to wrap ours.

Two fields make it yours to organize: `external_id` maps your own identifiers
onto Postrun resources, and `metadata` attaches your own context that you read
back and filter on. Both work on **profiles, posts, and media**.

**Concretely:** say you're building a tool for 200 agencies. Key each agency to a
profile with its own `external_id`, then mint each browser session a
[token](/authentication) scoped to just that profile. Their dashboard now lists
posts, uploads media, and reads ad results straight from Postrun — seeing only
their own data, with no database, no sync, and no backend of yours in between.

## Map with your own IDs

Set `external_id` to your own canonical id for a resource — your id for a client,
a campaign, a draft. It's **unique per account**, so you resolve your identifier
straight to a Postrun resource with no join table on your side. Reusing a value
that's already taken returns `409` [`conflict`](/errors/conflict) — pick a different one.

Filter by `external_id` on any resource that supports it — profiles, posts, and media:

<CodeGroup>
```ts @postrun/js (profiles)
import { profilesList } from "@postrun/js";

// Requires profiles:read
const { data } = await profilesList({
  client: postrun,
  query: { external_id: "client_acme" },
});
```

```ts @postrun/js (posts)
import { postsList } from "@postrun/js";

// Requires posts:read
const { data } = await postsList({
  client: postrun,
  query: { external_id: "campaign_q3_launch" },
});
```

```bash cURL
curl "https://api.postrun.ai/v1/profiles?external_id=client_acme" \
  -H "Authorization: Bearer $POSTRUN_TOKEN"

curl "https://api.postrun.ai/v1/posts?external_id=campaign_q3_launch" \
  -H "Authorization: Bearer $POSTRUN_TOKEN"
```
</CodeGroup>

This is also how your backend resolves "my client" to a profile when [minting a
token](/authentication#token-parameters) — look up the profile id by `external_id`,
then pass it into `profile_scope`.

## Attach your own context

`metadata` is a flat set of key/value pairs you own. Postrun stores it, returns
it on every read, and lets you filter by it — descriptive context, not a place to
branch authorization (it's writable from the browser, so never derive permissions
from it).

```json
{
  "metadata": {
    "tenant": "acme",
    "plan": "pro",
    "priority": 3,
    "archived": false
  }
}
```

Values are scalars, and the shape is capped to stay cleanly queryable. Exceeding
any limit returns `422` [`validation_failed`](/errors/validation_failed):

| Rule | Limit |
| --- | --- |
| Value types | `string`, `number`, or `boolean` — no arrays or nested objects |
| Keys | ≤ 50, each ≤ 40 characters |
| String values | ≤ 500 characters |
| Total size | ≤ 16 KiB (serialized JSON) |

## Filter by your context

List endpoints accept a `metadata` filter — a URL-encoded JSON object. The
underlying query is a Postgres `jsonb @>` containment check backed by a GIN
index, which means **exact-match only**: no operators, no ranges, no LIKE. Multiple
keys are ANDed.

<CodeGroup>
```ts @postrun/js
import { postsList } from "@postrun/js";

// Requires posts:read — pass the object directly, types preserved.
const { data } = await postsList({
  client: postrun,
  query: { metadata: { campaign: "summer-2026", priority: 3 } },
});
```

```bash cURL
# ?metadata={"campaign":"summer-2026","priority":3}
curl "https://api.postrun.ai/v1/posts?metadata=%7B%22campaign%22%3A%22summer-2026%22%2C%22priority%22%3A3%7D" \
  -H "Authorization: Bearer $POSTRUN_TOKEN"
```
</CodeGroup>

<Warning>
  Types must match exactly. A stored number `3` queried as the string `"3"` returns
  zero rows — no error, just an empty result. Keep a key's value type consistent
  across records.
</Warning>
