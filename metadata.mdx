---
title: "Metadata & external IDs"
description: "Map your own identifiers onto Postrun resources, and attach queryable context — no database of your own."
---

Postrun is the system of record for your social and ads domain, so you don't run
a parallel database to track it. Two fields make that work: `external_id` maps
your identifiers onto Postrun resources, and `metadata` attaches your own context
that you can read back and filter on. Both are available on **profiles, posts,
and media**.

## external_id

Set `external_id` to your own canonical id for a resource — your id for a client,
a campaign, a draft. It's **unique per account** and indexed, so you can resolve
your identifier straight to a Postrun resource without keeping a join table:

<CodeGroup>
```bash cURL
curl "https://api.postrun.ai/v1/profiles?external_id=client_acme" \
  -H "Authorization: Bearer $POSTRUN_SECRET_KEY"
```

```ts @postrun/js
import { profilesList } from "@postrun/js";

const { data } = await profilesList({
  client: postrun.client,
  query: { external_id: "client_acme" },
});
```
</CodeGroup>

Because it's unique, reusing a value that's already taken returns `409`
[`conflict`](/errors/conflict) — pick a different one. This is also what lets your
backend resolve "my client" to a profile when [minting a
token](/authentication#choosing-profiles), with no lookup of your own.

## metadata

`metadata` is a flat set of key/value pairs you own. Postrun stores it, returns
it on every read, and lets you filter by it — descriptive context, not a place to
branch authorization (it's writable from the browser, so never derive permissions
from it).

```json
{
  "metadata": {
    "tenant": "acme",
    "plan": "pro",
    "priority": 3,
    "archived": false
  }
}
```

Values are scalars, and the shape is capped to stay cleanly queryable:

| Rule | Limit |
| --- | --- |
| Value types | `string`, `number`, or `boolean` — no arrays or nested objects |
| Keys | ≤ 50, each ≤ 40 characters |
| String values | ≤ 500 characters |
| Total size | ≤ 16 KiB (serialized JSON) |

## Filtering by metadata

List endpoints accept a `metadata` filter — a URL-encoded JSON object. Matches
are **exact**, multiple keys are **ANDed**, and types must match (so `3` matches
the number `3`, not the string `"3"`):

<CodeGroup>
```bash cURL
# ?metadata={"campaign":"summer-2026","priority":3}
curl "https://api.postrun.ai/v1/posts?metadata=%7B%22campaign%22%3A%22summer-2026%22%2C%22priority%22%3A3%7D" \
  -H "Authorization: Bearer $POSTRUN_SECRET_KEY"
```

```ts @postrun/js
import { postsList } from "@postrun/js";

// Pass the object directly — no URL-encoding, and types are preserved.
const { data } = await postsList({
  client: postrun.client,
  query: { metadata: { campaign: "summer-2026", priority: 3 } },
});
```
</CodeGroup>

<Note>
  Keep a key's value type consistent across records — filtering matches the type
  exactly. The filter is exact-match only; there are no operators or ranges.
</Note>
