---
title: "Idempotency"
description: "Retry write requests safely, with no risk of the same action happening twice."
---

An idempotency key makes a write safe to retry. Attach a unique key to a request and Postrun runs that key **at most once** — retry as many times as you need and you'll get the original result back, not a second action.

## How it works

Send an `Idempotency-Key` header on any write request:

<CodeGroup>
```ts @postrun/js
import { postsCreate } from "@postrun/js";

await postsCreate({
  client: postrun,
  headers: { "Idempotency-Key": "1b9d6bcd-bbfd-4b2d-9b5d-ab8dfbbd4bed" },
  body: { /* ... */ },
});
```

```bash cURL
curl https://api.postrun.ai/v1/posts \
  -H "Authorization: Bearer $POSTRUN_TOKEN" \
  -H "Idempotency-Key: 1b9d6bcd-bbfd-4b2d-9b5d-ab8dfbbd4bed" \
  -H "Content-Type: application/json" \
  -d '{ "...": "..." }'
```
</CodeGroup>

The first request with a given key executes normally. Any later request with the
**same key and the same fingerprint** — the HTTP method, request path (including
path params), and request body — returns the original response byte-for-byte,
without running the action again.

Idempotency is **opt-in per endpoint**. Endpoints that support it are noted in the
[API reference](/api-reference); on any other endpoint the header is a no-op. A
frontend token can send `Idempotency-Key` on any write it's permitted to make.

<Note>
  Reads (`GET`) are already safe to repeat and don't need a key.
</Note>

## The key

You generate the key. It's any unique, opaque string of **1–255 characters** —
a version-4 UUID is the right default:

```ts
import { randomUUID } from "node:crypto";

const idempotencyKey = randomUUID();
```

One key represents one logical operation. Generate it **once**, before the first
attempt, and reuse that same value for every retry of that operation. The
durable pattern is to persist the key alongside whatever you're creating, so a
process that crashes and restarts retries with the same key rather than minting
a new one.

Keys are scoped to your account and retained for **~24h** from first use, then
expire — reusing one after that starts a fresh operation.

## What a retry returns

How a repeated key resolves depends on what happened the first time:

| Your retry | What you get back |
| --- | --- |
| Same key, same fingerprint, first request reached a **final result** — a success, or a deliberate error (e.g. validation, conflict) | That exact result, replayed. The action does not run again. |
| Same key, first request failed with an **unexpected 5xx**, where it's unclear whether the action took effect | The key is released — a retry with the same key runs the operation fresh. |
| Same key while the first request is **still running** | `409` [`idempotency_request_in_progress`](/errors/idempotency_request_in_progress). Wait 1–2 seconds, then retry the same key. |
| Same key, **different** method, path, or body | `422` [`idempotency_key_reused`](/errors/idempotency_key_reused). |
| Empty key, or longer than 255 characters | `400` [`idempotency_key_invalid`](/errors/idempotency_key_invalid). |

## When an attempt fails

Retry with the **same key** and exponential backoff — including on `409
idempotency_request_in_progress` (wait 1–2s first) and on `5xx` responses (the
key is released and the operation will re-run). If the first attempt already
reached a final result, you get that result back — a success stays one success, a
deliberate rejection stays the same error, and the action never runs twice. If it
failed with an unexpected error, the same key runs the operation again so it can
resolve into a real success or a final error.

<Warning>
  Never mint a new key to "get past" a failure. A new key is a new operation —
  that's exactly how duplicates happen. Always retry with the **same** key.
</Warning>

## Reusing a key with a different request

A key is bound to its exact fingerprint: HTTP method + request path (including path
params) + body. Field order in JSON doesn't matter — re-serializing the same body
still counts as the same request — but sending the same key to a different endpoint
or `:id`, or with a different body, returns `422`
[`idempotency_key_reused`](/errors/idempotency_key_reused). This catches a common
bug — a key accidentally shared across two different operations — before it does
damage.

The rule: **one key, one operation.** Don't recycle a key for a new action, and
don't change the body on a retry.

## Concurrency

If two requests carrying the same key arrive at once, exactly one executes; the
others receive `409`
[`idempotency_request_in_progress`](/errors/idempotency_request_in_progress)
while the first is in flight. Treat the `409` as "the first attempt is still
working" — wait 1–2 seconds, then retry the same key.
