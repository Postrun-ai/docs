---
title: "Idempotency"
description: "Retry write requests safely, with no risk of the same action happening twice."
---

Networks fail. A request times out, a connection drops, your job runner restarts
mid-call — and you're left not knowing whether the write went through. Retrying
blindly risks doing it twice: a double post, a duplicated campaign, a budget
changed twice.

An idempotency key removes that uncertainty. You attach a unique key to a write,
and Postrun runs that key **at most once**. Retry as many times as you need —
you'll get the original result back, not a second action.

## How it works

Send an `Idempotency-Key` header on any write request:

<CodeGroup>
```ts @postrun/js
import { postsCreate } from "@postrun/js";

await postsCreate({
  client: postrun.client,
  headers: { "Idempotency-Key": "1b9d6bcd-bbfd-4b2d-9b5d-ab8dfbbd4bed" },
  body: { /* ... */ },
});
```

```bash cURL
curl https://api.postrun.ai/v1/posts \
  -H "Authorization: Bearer $POSTRUN_SECRET_KEY" \
  -H "Idempotency-Key: 1b9d6bcd-bbfd-4b2d-9b5d-ab8dfbbd4bed" \
  -H "Content-Type: application/json" \
  -d '{ "...": "..." }'
```
</CodeGroup>

The first request with a given key executes normally, and Postrun records its
outcome. Any later request with the **same key and the same payload** returns
that recorded outcome — the original response, byte for byte — without running
the action again.

Idempotency is **opt-in per request**. Omit the header and the endpoint behaves
exactly as it would otherwise. Send it whenever a retry is possible — which, over
an unreliable network, is every write worth protecting.

<Note>
  Reads (`GET`) are already safe to repeat and don't need a key. Idempotency
  applies to write requests; each write endpoint notes support in its
  [API reference](/api-reference).
</Note>

## The key

You generate the key. It's any unique, opaque string of **1–255 characters** —
a version-4 UUID is the right default:

```ts
import { randomUUID } from "node:crypto";

const idempotencyKey = randomUUID();
```

One key represents one logical operation. Generate it **once**, before the first
attempt, and reuse that same value for every retry of that operation. The
durable pattern is to persist the key alongside whatever you're creating, so a
process that crashes and restarts retries with the same key rather than minting a
new one.

Keys are scoped to your account and retained for **24 hours** from first use,
then expire — reusing one after that starts a fresh operation.

## What a retry returns

How a repeated key resolves depends entirely on what happened the first time:

| Your retry | What you get back |
| --- | --- |
| Same key, same payload, first request reached a **final result** — a success, or an error Postrun deliberately returned (e.g. validation, conflict) | That exact result, replayed. The action does not run again. |
| Same key, first request failed with an **unexpected error**, where it's unclear whether the action took effect | The key isn't pinned to that failure — a retry with the same key runs the operation (see below). |
| Same key while the first request is **still running** | `409` [`idempotency_request_in_progress`](/errors/idempotency_request_in_progress). Wait and retry. |
| Same key, **different** payload | `422` [`idempotency_key_reused`](/errors/idempotency_key_reused). |
| Empty key, or longer than 255 characters | `400` [`idempotency_key_invalid`](/errors/idempotency_key_invalid). |

## When an attempt fails

Whatever the failure, retry with the **same key** and exponential backoff. If the
first attempt already reached a final result, you get that result back — a success
stays one success, a deliberate rejection stays the same error, and the action
never runs twice. If it failed with an unexpected error where the outcome is
unclear, the same key runs the operation again so it can resolve into a real
success or a real, final error. Either way, you can't create a duplicate, and you
can't get stuck behind a transient fault.

<Warning>
  Always retry a failed write — including `409`
  [`idempotency_request_in_progress`](/errors/idempotency_request_in_progress) —
  with the **same** key and backoff. Never mint a new key to "get past" a
  failure: a new key is a new operation, and that's exactly how duplicates
  happen.
</Warning>

## Reusing a key with a different request

A key is bound to the exact request that first used it. Field order in a JSON body
doesn't matter — re-serializing the same request still counts as the same request
— but a different payload does: send the same key with one and Postrun rejects it
with `422` [`idempotency_key_reused`](/errors/idempotency_key_reused) rather than
guessing which request you meant. This catches a common bug — a key accidentally shared
across two different operations — before it does damage.

The rule that follows: **one key, one operation.** Don't recycle a key for a new
action, and don't change the body on a retry.

## Concurrency

If two requests carrying the same key arrive at once, exactly one executes; the
others receive `409`
[`idempotency_request_in_progress`](/errors/idempotency_request_in_progress)
while the first is in flight. There's no window where the same key runs twice in
parallel. Treat the `409` as "the first attempt is still working" — wait briefly,
then retry the same key.

## Best practices

<Steps>
  <Step title="One key per logical operation">
    Generate a fresh UUID for each distinct action. Never share a key across
    different operations.
  </Step>
  <Step title="Persist the key before you send it">
    Store the key with the record you're creating. If your process restarts
    mid-flight, retry with the saved key — that's what makes the retry safe.
  </Step>
  <Step title="Retry 5xx and 409 with the same key">
    Use exponential backoff, and keep the key identical across attempts. A new
    key on a retry defeats the purpose.
  </Step>
  <Step title="Keep keys opaque">
    A random UUID is ideal. Don't encode meaningful or sensitive data into the
    key — it's an identifier, not a payload.
  </Step>
</Steps>
