title: "Authentication"
description: "A secret key for your backend, scoped tokens for the browser, with authorization built into every token."
---

There are two credentials: your backend holds a secret `pr_` key and mints short-lived scoped tokens; the browser holds only the token and calls Postrun directly. Every request carries one as `Authorization: Bearer <credential>`.

| Credential | Access | Where it can run |
| --- | --- | --- |
| **Secret key** (`pr_…`) | Your whole account | Backend only — never the browser |
| **Scoped token** | Only the profiles and scopes you grant it | Anywhere — browser or backend |

## End-to-end flow

**Step 1 — your backend mints a token.** Call `POST /v1/tokens` with your secret key. Scope it to exactly what this session needs:

<CodeGroup>
```ts Backend route (Next.js example)
// app/api/token/route.ts — server-side only
import { NextResponse } from "next/server";

export async function GET() {
  const res = await fetch("https://api.postrun.ai/v1/tokens", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${process.env.POSTRUN_SECRET_KEY}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      profile_scope: { type: "external_id", values: ["client_acme"] },
      scopes: ["posts:read", "posts:write", "media:write"],
      ttl_seconds: 900,
    }),
  });
  const { token } = await res.json();
  return NextResponse.json({ token });
}
```

```bash cURL
curl https://api.postrun.ai/v1/tokens \
  -H "Authorization: Bearer $POSTRUN_SECRET_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "profile_scope": { "type": "external_id", "values": ["client_acme"] },
    "scopes": ["posts:read", "posts:write", "media:write"],
    "ttl_seconds": 900
  }'
```
</CodeGroup>

**Step 2 — the browser fetches a token from your route.** Define `getToken` once:

```ts
async function getToken(): Promise<string> {
  const res = await fetch("/api/token");
  const { token } = await res.json();
  return token;
}
```

**Step 3 — wire it into the SDK.** Pass `getToken` to either client:

<CodeGroup>
```ts @postrun/js
import { createPostrunClient, postsList } from "@postrun/js";

const postrun = createPostrunClient({ getToken });

const { data } = await postsList({ client: postrun, query: { limit: 20 } });
```

```tsx @postrun/react
import { PostrunProvider } from "@postrun/react";

export default function App() {
  return (
    <PostrunProvider getToken={getToken}>
      {/* hooks work anywhere inside */}
    </PostrunProvider>
  );
}
```
</CodeGroup>

<Note>
  The React provider re-calls `getToken` automatically when a token is expired
  or a request returns `401`. See the [React SDK](/react/get-started) for the
  full provider setup and hooks.
</Note>

## Token parameters

`POST /v1/tokens` accepts:

| Parameter | Description |
| --- | --- |
| `profile_scope` | Which profiles the token may act on (see below) |
| `scopes` | Which actions the token may take (see below) |
| `ttl_seconds` | Lifetime in seconds — min `60`, max `1800`, default `900` |

**`profile_scope`** — one of three selectors:

| Selector | Cap | Use it when |
| --- | --- | --- |
| `{ "type": "profile_ids", "values": [...] }` | ≤ 100 ids | You have Postrun profile ids. |
| `{ "type": "external_id", "values": [...] }` | ≤ 50 | You key profiles by [your own id](/storage). Resolved per request, so it covers profiles created after mint. |
| `{ "type": "all" }` | — | The token may act on every profile in the account. |

Exceeding a cap returns `422` [`validation_failed`](/errors/validation_failed). Minting with a profile id that doesn't exist in your account returns `422` `profile_scope_invalid`.

## Token errors

| Status | Code | Meaning |
| --- | --- | --- |
| `401` | [`unauthorized`](/errors/unauthorized) | Token is expired or invalid. Re-mint and retry. |
| `403` | [`forbidden`](/errors/forbidden) | Token lacks the required scope. Mint with the right scopes. |
| `404` | [`not_found`](/errors/not_found) | Profile is outside the token's `profile_scope`. The token can't confirm the profile exists. |
| `422` | `profile_scope_invalid` | Minted with a profile id that doesn't belong to your account. |

There is no `token_expired` code — expiry surfaces as `401 unauthorized`.

## Scopes

A token's authorization is two independent choices: **whose data** (`profile_scope`) and **which actions** (scopes).

Coarse: one read, one write, and one delete per resource. `:write` grants the matching `:read`, so you never list both. `:delete` is an **independent** third action — `:write` does **not** include it, so you can grant full read-and-write yet withhold the power to destroy; `:delete` grants the matching `:read` (you can always see what you can delete):

| Resource | Read | Write | Delete |
| --- | --- | --- | --- |
| Profiles | `profiles:read` | `profiles:write` | `profiles:delete` |
| Connections | `connections:read` | `connections:write` | `connections:delete` |
| Posts | `posts:read` | `posts:write` | `posts:delete` |
| Media | `media:read` | `media:write` | `media:delete` |
| Ads | `ads:read` | `ads:write` | `ads:delete` |

Mint the narrowest set a session needs.

## Secret keys

Create a `pr_` key in the [dashboard](https://app.postrun.ai). It has **full access to your whole account**, so keep it server-side, store it in an environment variable, and never ship it to a browser or commit it.

Use the secret key for two things:

1. **Minting scoped tokens** — the flow above.
2. **Money and structural writes** — operations a browser token can't reach (see below).

## What only a secret key can do

Some writes are **secret-key only** — they don't accept a scoped token at all, regardless of its scopes. Money and structural writes live here: anything that changes ad budget or spend, for example, can only be called from your backend — like a Stripe publishable key that can tokenize a card but can't issue a refund.

Whether an endpoint accepts a token is stated in its [API reference](/api-reference) under _Authorizations_. If it lists only the secret key, a browser token can't call it.
