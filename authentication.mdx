title: "Authentication"
description: "A secret key for your backend, scoped tokens for the browser, with authorization built into every token."

To let a browser talk to a third-party API safely, you normally stand up a
backend: it holds the secret, proxies every call, and on each request decides
whether _this_ user may touch _that_ data. That backend is plumbing you build and
maintain. With Postrun, you don't need it.

There are two credentials, and every request carries one as
`Authorization: Bearer <credential>`:

| Credential | Access | Where it can run |
| --- | --- | --- |
| **Secret key** (`pr_ŌĆ”`) | Your whole account | Backend only ŌĆö never the browser |
| **Scoped token** | Only the profiles and scopes you grant it | Anywhere ŌĆö browser or backend |

Your backend mints a short-lived **scoped token** from the secret key and hands it
to the browser. The browser then calls Postrun **directly** ŌĆö no proxy in between.

A scoped token isn't browser-only ŌĆö it works anywhere, and because it's limited
and short-lived, it's the safer choice for server-side calls too. Reach for the
secret key only to mint tokens and to make money or structural writes, which a
token can't do.

<Note>
  Every request answers two questions. **Authentication** ŌĆö who is calling? (the
  bearer credential.) **Authorization** ŌĆö what may they do? (for a token, the
  scopes and profiles baked into it.) Postrun enforces both on every request, so
  there are no permission checks left for you to write.
</Note>

## Scoped tokens ŌĆö the recommended path

A scoped token grants the least privilege that does the job, so it's the
credential to reach for by default. It's the only one that can run in a browser ŌĆö
so any user-facing app (a dashboard, a composer, a mobile app) must use one ŌĆö and
it's the safer choice on the server too. What this gives you:

- **No proxy backend.** The browser lists posts, reads ad results, uploads media,
  and publishes ŌĆö straight to Postrun. You don't write routes that forward calls
  and re-check permissions.
- **The token carries its own permissions.** A token is locked to specific
  **profiles** (whose data) and **scopes** (which actions), and Postrun checks
  both on every request. You decide what a token can reach at mint time, not in
  access-control code you maintain.
- **Safe by default.** The secret key never reaches the browser. A token is
  short-lived and scoped, and can't perform money or structural writes (more
  below). A leaked token expires on schedule and can only do what you scoped it
  to.
- **One network hop.** The browser reaches Postrun directly, with no proxy
  round-trip through your backend.

### Mint a token

Minting happens on your **backend**, which holds the secret key. Scope it to
exactly what the session needs.

<CodeGroup>
```ts @postrun/js
import { createPostrunServer } from "@postrun/js/server";

// Backend only ŌĆö holds your secret pr_ key, never the browser.
const postrun = createPostrunServer({
  secretKey: process.env.POSTRUN_SECRET_KEY!,
});

const { token, expires_at } = await postrun.tokens.mint({
  profile_scope: { type: "external_id", values: ["client_acme"] },
  scopes: ["posts:read", "posts:write", "media:write"],
  ttl_seconds: 900,
});
```

```bash cURL
curl https://api.postrun.ai/v1/tokens \
  -H "Authorization: Bearer $POSTRUN_SECRET_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "profile_scope": { "type": "external_id", "values": ["client_acme"] },
    "scopes": ["posts:read", "posts:write", "media:write"],
    "ttl_seconds": 900
  }'
```
</CodeGroup>

### Scope it on two axes

A token's authorization is two independent choices: **whose data** it can reach,
and **which actions** it can take.

**Profiles ŌĆö whose data.** `profile_scope` picks the profiles the token may act
on:

| Selector | Use it when |
| --- | --- |
| `{ "type": "profile_ids", "values": [...] }` | You have Postrun profile ids. |
| `{ "type": "external_id", "values": [...] }` | You key profiles by [your own id](/storage). Resolved per request, so it also covers profiles created after the token was minted. |
| `{ "type": "all" }` | The token may act on every profile in the account. |

**Scopes ŌĆö which actions.** Coarse: one read, one write, and one delete per
resource. A `:write` grants the matching `:read`, so you never list both. `:delete`
is an **independent** third action ŌĆö `:write` does **not** include it, so you can
grant full read-and-write yet withhold the power to destroy; `:delete` grants the
matching `:read` (you can always see what you can delete):

| Resource | Read | Write | Delete |
| --- | --- | --- | --- |
| Profiles | `profiles:read` | `profiles:write` | `profiles:delete` |
| Connections | `connections:read` | `connections:write` | `connections:delete` |
| Posts | `posts:read` | `posts:write` | `posts:delete` |
| Media | `media:read` | `media:write` | `media:delete` |
| Ads | `ads:read` | `ads:write` | `ads:delete` |

Mint the narrowest set a session needs. Ask a token for a profile it wasn't
scoped to and you get `404` [`not_found`](/errors/not_found), not `403` ŌĆö so a
token can't discover profiles outside its scope.

### Lifetime

`ttl_seconds` sets how long the token lives ŌĆö between 60 and 1800, defaulting to
900 (15 minutes). Tokens expire instead of being revoked: there's no revocation
list to manage, so mint narrowly and re-mint as the session needs. The response
returns the resolved `expires_at`.

### Use it in the browser

The browser holds only the token, and calls Postrun with it as the bearer ŌĆö same
API, narrowed to the token's scopes and profiles.

<CodeGroup>
```ts @postrun/js
import { createPostrunClient, postsList } from "@postrun/js";

// Browser ŌĆö `getToken` fetches the token your backend minted.
const postrun = createPostrunClient({ getToken });

const { data } = await postsList({ client: postrun, query: { limit: 20 } });
```

```bash cURL
curl https://api.postrun.ai/v1/posts \
  -H "Authorization: Bearer $POSTRUN_FRONTEND_TOKEN"
```
</CodeGroup>

<Note>
  Building a UI? The [React SDK](https://www.npmjs.com/package/@postrun/react)
  takes `getToken` once in a provider and gives you hooks that fetch, cache, and
  refresh for you ŌĆö so you don't manage the token yourself.
</Note>

## Secret keys

Create a `pr_` key in the [dashboard](https://app.postrun.ai). It has **full
access to your whole account**, so treat it like a password: keep it server-side,
store it in an environment variable, and never ship it to a browser or commit it.

A secret key belongs on your **backend**, and ideally does just two things:

1. **Mint scoped tokens** ŌĆö the flow above, for anything user-facing.
2. **Money and structural writes** ŌĆö operations a browser token can't reach (next
   section), called from your own server.

For everything else ŌĆö including server-side calls that don't need full account
access ŌĆö prefer a scoped token. It keeps the secret key in fewer places and
limits what a leaked credential can do.

## What only a secret key can do

Some writes are **secret-key only** ŌĆö they don't accept a scoped token at all, no
matter its scopes. **Money and structural writes** live here: anything that
changes ad budget or spend, for example, can only be called from your backend. So
a leaked or over-scoped browser token still can't touch them ŌĆö like a Stripe
publishable key that can tokenize a card but can't issue a refund.

Whether an endpoint accepts a token is stated in its
[API reference](/api-reference) under _Authorizations_. If it lists only the
secret key, a browser token can't call it.

## Which credential?

| You're buildingŌĆ” | Use |
| --- | --- |
| A user-facing UI (dashboard, composer, app) | A **scoped token** ŌĆö required; the secret key can't run in a browser |
| A server-side call that needs only some profiles or actions | A **scoped token** ŌĆö least privilege, safer if leaked |
| A server-side integration that needs full account access | The **secret key** |
| A money or structural write (ad budget, spend) | The **secret key**, from your backend |
| Minting tokens | The **secret key**, from your backend |
