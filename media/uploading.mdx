---
title: "Uploading media"
description: "Hand the SDK a file; it uploads it resumably and Postrun processes it into a platform-ready rendition for each target."
---

You don't manage the upload yourself. The SDK takes a file, uploads it as a
**resumable multipart upload**, and Postrun processes it into a per-platform
rendition in the background — you get back an asset you can attach to a post.

## Upload a file

<CodeGroup>
```tsx @postrun/react icon="react"
import { useMediaUpload } from "@postrun/react";

function Uploader({ profileId }) {
  const { add, items, isUploading } = useMediaUpload();

  return (
    <input
      type="file"
      multiple
      disabled={isUploading}
      onChange={(e) =>
        add(e.target.files, { profileId, targets: ["x", "instagram"] })
      }
    />
  );
  // each entry in `items` carries { file, status, progress, media }
}
```

```ts @postrun/js icon="square-js"
import { mediaCreate, uploadFile } from "@postrun/js";

// 1. Create the asset (and get its upload session)
const { data: media } = await mediaCreate({
  client: postrun.client,
  body: { profile_id: "prof_3b9d6bcd", targets: ["x", "instagram"] },
});

// 2. Upload the file — resumable multipart, handled for you
if (media.upload) {
  await uploadFile(file, {
    client: postrun.client,
    mediaId: media.id,
    session: media.upload,
    onProgress: (fraction) => console.log(Math.round(fraction * 100)),
  });
}
```
</CodeGroup>

The file type is detected from the bytes — you never declare it. Chunking, parallel
parts, retries, and resuming after a dropped connection all happen inside the SDK.
The React hook also paces multiple files for you and exposes each one's live state —
see [React SDK → Media](/react/media) for progress, previews, and managing assets.

## Targets

`targets` lists the platforms you intend to publish to; Postrun validates and
renders a rendition for each. Targets are the post platforms (`x`, `instagram`,
`tiktok`, `linkedin`, `facebook_page`) and the ad target `google_ads`.

You can always add more supported platforms later by
[updating the asset](/api-reference/media/update-media) — Postrun renders each new
target on demand.

## Import from a URL

If the bytes already live at a public `https` URL, skip the upload entirely: pass a
`source_url` and Postrun fetches, validates, transforms, and stores them.

```ts @postrun/js icon="square-js"
import { mediaCreate } from "@postrun/js";

const { data: media } = await mediaCreate({
  client: postrun.client,
  body: {
    profile_id: "prof_3b9d6bcd",
    source_url: "https://example.com/launch.mp4",
    targets: ["x", "instagram", "tiktok"],
  },
});
// nothing to upload — the bytes are fetched and processed in the background
```

## Wait until it's ready

Processing runs asynchronously. The React hook tracks it for you — each item moves
`uploading → processing → ready` (or `failed`), and `progress` drives a bar. Outside
React, `await` the asset with `waitForMedia` — no poll loop to write:

```ts @postrun/js icon="square-js"
import { waitForMedia } from "@postrun/js";

const asset = await waitForMedia(media.id, { client: postrun.client });
// asset.status is "ready" (or "failed"); asset.per_platform holds the renditions
```

`waitForMedia` polls until the asset settles. A `failed` asset resolves too — read
its `per_platform` errors; only a timeout or an aborted `signal` rejects. Pass
`onPoll` to surface `progress.percent`, or `timeout` / `signal` to bound the wait:

```ts @postrun/js icon="square-js"
const asset = await waitForMedia(media.id, {
  client: postrun.client,
  onPoll: (m) => console.log(m.progress.percent),
  timeout: 5 * 60_000,
  signal: controller.signal,
});
```

When it's `ready`, each target's rendition is in `per_platform` — its URL,
dimensions, and any per-platform findings. Reading that result is covered next.
