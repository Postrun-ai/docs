The asset shape is identical from `mediaCreate`, `waitForMedia`, `useMedia`, or `GET /v1/media/{id}`. When it **settles** (`status` is `ready` or `failed`), it carries a typed result for each platform you targeted in **`per_platform`** ‚Äî one entry per target.

## The per-platform map

Each entry has the rendition `url` (use it when you post to that platform), its `width` / `height` / `bytes`, and two lists of findings ‚Äî `warnings` and `errors`:

```json
{
  "id": "med_9f2a",
  "status": "ready",
  "progress": { "stage": "done", "percent": 100 },
  "per_platform": {
    "x": {
      "status": "ready",
      "url": "https://media.postrun.ai/med_9f2a/x.mp4",
      "width": 1280,
      "height": 720,
      "bytes": 4185000,
      "warnings": [
        {
          "code": "media_format_recompressed",
          "display_error": "This video was re-encoded to meet X's requirements.",
          "message": "Re-encoded HEVC to H.264 for X."
        }
      ],
      "errors": []
    },
    "instagram": {
      "status": "failed",
      "url": null,
      "warnings": [],
      "errors": [
        {
          "code": "media_aspect_ratio_unsupported",
          "display_error": "This image's aspect ratio isn't supported on Instagram.",
          "message": "Aspect ratio 3:1 is outside the allowed range.",
          "allowed": ["4:5 ‚Äì 1.91:1"],
          "got": "3:1"
        }
      ]
    }
  }
}
```

`asset.status: "ready"` means processing finished ‚Äî it does **not** mean every platform succeeded. An asset can be `ready` while one or more `per_platform` entries are `failed` (the original processed fine, but a platform rejected it). Always check each `per_platform[target].status` before using its `url`. While `status` is `processing`, each entry exists but its `url` is `null` and `errors` / `warnings` are empty ‚Äî read them only once the asset has settled.

A top-level failure (corrupt file, unrecognized format, or unreachable `source_url`) sets `status` to `failed` at the asset level with the top-level `error` populated. `per_platform` is empty ‚Äî no platform entry exists when the original couldn't be processed at all:

```json
{
  "id": "med_7c1b",
  "status": "failed",
  "progress": { "stage": "done", "percent": 100 },
  "error": {
    "code": "media_fetch_failed",
    "display_error": "The URL couldn't be fetched.",
    "message": "GET https://example.com/launch.mp4 returned 403."
  },
  "per_platform": {}
}
```

## Problems surface in three places

- **`per_platform[target].errors`** ‚Äî **blocking** for that target. The file can't be published there (wrong aspect ratio, unsupported codec, too large). The entry's `status` is `failed` and its `url` is `null`.
- **`per_platform[target].warnings`** ‚Äî **non-blocking**. Postrun adjusted the file to fit (re-encoded, recompressed, downscaled); the rendition is still `ready` and usable. Log it or show it in a developer UI; it needs no user action.
- **`error`** (top level, asset-wide) ‚Äî the **original itself** is unusable (corrupt file, unrecognized format, or unreachable `source_url`). No platform can succeed. It's `null` otherwise.

<Note>
  Gate your publish on `per_platform[target].status === "ready"` and `errors.length === 0`. Never key on `warnings`.
</Note>

## Read it

<CodeGroup>
```tsx @postrun/react icon="react"
import { useMedia } from "@postrun/react";

function XRendition({ id }) {
  const { data: media } = useMedia(id); // auto-refetches while processing

  const x = media?.per_platform.x;
  if (!x) return <p>Loading‚Ä¶</p>;
  if (x.status === "failed") return <p>{x.errors[0]?.display_error}</p>;
  if (x.status !== "ready" || !x.url) return <p>Processing‚Ä¶</p>;

  return <video src={x.url} controls />;
}
```

```ts @postrun/js icon="square-js"
import { waitForMedia } from "@postrun/js";

const asset = await waitForMedia(media.id, { client: postrun.client });

const x = asset.per_platform.x;
if (x?.status === "ready" && x.url) {
  // use x.url when you post to X
} else if (x?.status === "failed") {
  // branch on the typed code; show the user x.errors[0].display_error
  console.log(x.errors[0]?.code, x.errors[0]?.display_error);
}
```
</CodeGroup>

## Common error codes

| Code | Kind | Notes |
| --- | --- | --- |
| [`media_too_large`](/errors/media_too_large) | error | File exceeds 1 GB (non-video). |
| [`video_too_large`](/errors/video_too_large) | error | Video exceeds 4 GB. |
| [`media_aspect_ratio_unsupported`](/errors/media_aspect_ratio_unsupported) | error | Image aspect ratio outside the platform's allowed range. |
| [`video_aspect_unsupported`](/errors/video_aspect_unsupported) | error | Video aspect ratio outside the platform's allowed range. |
| [`video_codec_unsupported`](/errors/video_codec_unsupported) | error | Source codec can't be re-encoded for this target. |
| [`video_duration_exceeds_max`](/errors/video_duration_exceeds_max) | error | Clip is longer than the platform (or account) allows. |
| `video_duration_too_short` | error | Clip is shorter than the platform's minimum. |
| [`media_resolution_too_low`](/errors/media_resolution_too_low) | error | Dimensions are below the platform's minimum. |
| `video_too_small` | error | Video dimensions are below the platform's minimum. |
| `media_format_unsupported` | error | File format can't be converted to anything the platform accepts. |
| `media_unsupported` | error | File kind not supported for this target (e.g. a video sent to an image-only target). |
| `media_fetch_failed` | error | `source_url` couldn't be fetched; asset settles as a top-level failure. |
| [`media_format_recompressed`](/errors/media_format_recompressed) | warning | File was re-encoded or recompressed to fit. Rendition is still ready. |
| `media_resolution_downscaled` | warning | Image was resized down to the platform's dimension limit. Rendition is still ready. |

## Error fields

`warnings`, `errors`, and the asset-wide `error` share the same shape:

| Field | Use it for |
| --- | --- |
| `code` | Branch on it ‚Äî a stable, typed value documented in [Errors](/errors/overview). |
| `display_error` | A short, friendly line you can show your user as-is. |
| `message` | Developer-grade detail for logs. Not for end users. |
| `allowed` / `got` | The accepted range and what the file actually was, on size/format findings. |

## Attach it to a post

Reference the asset's `id` in a post variant's `media` as `{ media_id: asset.id }` ‚Äî the publish path picks the matching `per_platform` rendition for each channel automatically. See [Posts](/react/posts).
